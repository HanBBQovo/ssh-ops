# 安全说明

## 策略模型

`sshctl exec` 默认会应用 denylist，拦截明显危险的命令模式，例如：

- `rm -rf /`
- `rm -rf *`
- `mkfs`
- `dd ... of=/dev/...`
- `shutdown`
- `reboot`
- `halt`
- `poweroff`
- fork bomb

用户也可以通过配置文件中的：

- `policy.allow_patterns`
- `policy.deny_patterns`

进一步收紧或放宽规则。

## 配置管理也应走最小变更

使用 `sshctl config ...` 时，推荐原则仍然是最小变更：

- 新增一个 host 时，只补当前任务需要的最小字段
- 更新配置时，只改明确需要变更的字段
- 删除和重命名主机前，先确认不会影响现有脚本或团队约定

换句话说，“不用手改 YAML”不等于“可以批量无差别改配置”。

## 推荐的安全操作方式

- 先执行只读命令，例如 `pwd`、`ls`、`df -h`、`stat`、`tail -n 200`
- 如果要写入，尽量拆成更小、更明确的动作
- 优先使用 SFTP 传文件，再执行短小的验证命令
- 能用 `--workdir` 时，尽量不要在命令里拼很长的 `cd && ...`
- 对生产机优先做检查，再做变更

## 遇到这些情况要显式提示风险

如果任务涉及以下内容，应先向用户说明风险，再继续：

- 重启、关机、停止系统服务
- 改写磁盘、文件系统、分区
- 批量删除文件
- 修改生产配置但没有回滚说明
- 输出或暴露敏感信息
- 启动不可控的后台长任务

## 不应该做的事

- 不要绕过 `policy_denied`
- 不要擅自切换到 `insecure_ignore`
- 不要把明文密码、私钥内容贴进输出
- 不要把大量原始日志整段塞进最终回答
- 不要泄露 `sshctl config show --reveal-secrets` 看到的秘密值
- 不要为了图快就把所有 host 都改一遍

## 一个更稳妥的执行顺序

1. 识别 host id
2. 校验配置
3. 先跑只读命令确认状态
4. 再执行最小范围的写操作
5. 执行后补一个验证命令
6. 汇报结果和潜在风险
